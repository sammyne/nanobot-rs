# 需求

## 目标与背景

当前 `catchup-hkuds` agent 的输入仅支持两个 commit SHA 作为范围边界。实际使用中，上游 HKUDS/nanobot 仓库有明确的 tag 版本（如 `v0.1.4`、`v0.1.5`、`v0.2.0`），用户更自然的操作是按 tag 指定范围（如 `v0.1.4 v0.2.0`），而不是手动查找 tag 对应的 commit SHA。

不做这个需求的影响：用户每次都需要先手动执行 `git rev-parse <tag>` 获取 SHA，再粘贴给 agent，增加了不必要的操作步骤。

## 方案比较（强制）

### 方案 1: 扩展输入格式描述（最小可行版）

- 思路: 修改 agent 的输入格式说明，将 "commit SHA" 改为 "commit 引用（SHA 或 tag）"。由于 `git log ref1..ref2` 本身就支持 tag，核心工作流的 git 命令无需改动。主要调整输入说明、增加 tag 验证步骤、更新输出格式以展示原始引用。
- 优点: 改动最小，利用 git 原生能力，无需额外逻辑
- 缺点: 依赖 agent（LLM）正确理解并执行指令，无编程层面的保障
- 工作量估算: S

### 方案 2: 增加显式 tag 解析与双格式展示（理想架构）

- 思路: 在方案 1 基础上，增加一个显式的"引用解析"步骤：用 `git rev-parse` 将输入的 tag 解析为 SHA，在输出中同时展示 tag 名和对应 SHA。还增加输入验证（检查引用是否存在于 upstream）。
- 优点: 输出信息更完整（tag + SHA 双重标识），有明确的验证步骤防止无效输入
- 缺点: 增加了一个额外步骤，略微增加复杂度
- 工作量估算: S

### 推荐

推荐方案 2。工作量同为 S，但增加了输入验证和更丰富的输出信息，对用户体验有明显提升。显式解析步骤也让 agent 的行为更可预测。

## 功能需求列表

### 核心功能

- 输入格式扩展：支持 commit SHA、git tag、或两者混合作为范围边界（如 `v0.1.4 v0.2.0`、`v0.1.4 abc1234`、`abc1234 v0.2.0`）
- 输入验证：检查提供的引用（tag 或 SHA）是否存在于 upstream remote
- 引用解析：将 tag 解析为对应的 commit SHA，用于后续 git log 操作
- 输出格式更新：范围显示同时展示原始引用和解析后的 SHA（当输入为 tag 时）

### 扩展功能

- 无

## 非功能需求

- **可维护性**：agent 指令清晰，步骤编号连贯，新增步骤不破坏现有流程的逻辑顺序
- **兼容性**：保持对纯 SHA 输入的完全向后兼容

## 边界与不做事项

- 不支持分支名作为输入（仅 SHA 和 tag）
- 不修改输出文件名或整体结构
- 不改变 PR 分析逻辑（步骤 4-8 的核心流程不变）

## 假设与约束

- **技术假设**：上游 tag 已通过 `git fetch upstream --tags` 拉取到本地
- **环境约束**：agent 运行环境已配置 git 和 gh CLI

## 待确认事项

- 无
