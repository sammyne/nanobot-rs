# 需求

## 目标与背景

将上游 PR #2733 的 agent 运行时加固功能迁移到 Rust 版。当前 Rust 版 `re_act` 循环是简单的串行执行，没有上下文治理、检查点、大结果持久化、工具批处理等机制。长时间运行任务（如大规模重构）容易触发上下文溢出、工具结果过大、中断后状态丢失等问题。

**现状不足**：
- 工具结果无大小限制，大结果直接发给 LLM 可能超出上下文窗口
- 没有上下文窗口监控，历史消息无限增长
- 中断后无法恢复已完成的工具调用
- 所有工具串行执行，只读工具无法并行
- 没有重复外部查询节流

## 方案比较（强制）

### 方案 1: 分阶段逐步迁移（最小可行版）✅ 已选定

- 思路: 按依赖关系拆分为 7 个独立子任务，按优先级逐步实现
- 优点: 每步可独立验证和合入，风险可控
- 缺点: 需要多轮迭代
- 工作量估算: L（总计），每个子任务 S-M

### 方案 2: 一次性全量迁移（理想架构）

- 思路: 一个大 PR 包含所有 7 个组件
- 优点: 一步到位
- 缺点: 改动巨大，review 困难，回归风险高
- 工作量估算: XL

### 推荐

方案 1。按推荐顺序逐步实现。

## 功能需求列表

### 子任务 A: prepare_call 验证分离（难度: 极小）

从 `ToolRegistry::execute()` 中提取验证逻辑到 `prepare_call()` 方法，在执行前先验证工具名和参数。

### 子任务 B: 大工具结果持久化（难度: 小）

当工具结果超过 `max_tool_result_chars`（默认 16000）时，将完整结果写入磁盘文件，替换为包含预览和文件路径的引用字符串。

### 子任务 C: 跨模型消息兼容（难度: 小）

- 避免连续相同角色消息（合并内容）
- 确保 OpenAI 兼容 provider 不发送 thinking 等非标准字段

### 子任务 D: 风险感知工具批处理（难度: 小-中）

- Tool trait 新增 `read_only()` 和 `exclusive()` 方法
- 只读工具并行执行，副作用工具串行，shell 独占

### 子任务 E: 上下文窗口治理（难度: 中）

- 每次 LLM 调用前检查 token 预算
- 超出时裁剪历史消息（保持工具调用/结果边界完整）
- 大工具结果预算控制

### 子任务 F: 近期轮次检查点（难度: 中）

- 工具执行前保存检查点到 session metadata
- 中断恢复时还原已完成的工具结果
- 重叠去重

### 子任务 G: Provider 流式看门狗（难度: 大）

- 流式响应空闲超时检测
- 持久重试模式
- 重试等待期间心跳回调
- **注意**: 依赖流式支持，当前 Rust 版未实现流式，延后

## 非功能需求

- 所有加固层 fail-open：加固逻辑自身的错误不应导致 agent 循环崩溃
- 现有测试全部通过

## 边界与不做事项

- 子任务 G（流式看门狗）延后，等流式支持实现后再做
- 不修改 provider 的 `chat()` 接口（保持非流式）

## 假设与约束

- 按推荐顺序实现：A → B → C → D → E → F（G 延后）
- 每个子任务可独立提交和合入
