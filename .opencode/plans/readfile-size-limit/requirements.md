# 需求

## 目标与背景

对应上游 HKUDS/nanobot PR #1511（#21），为 ReadFileTool 添加文件大小限制，防止读取大文件导致 OOM 崩溃。

**现状问题**：`ReadFileTool::execute()` 在获取 metadata 后直接调用 `fs::read_to_string()`，无任何大小检查。如果 agent 尝试读取多 GB 文件（如日志、数据库转储），进程会因内存耗尽崩溃。

## 方案比较（强制）

### 方案 1: 两阶段检查（最小可行版）

- 思路: 在 `read_to_string` 前检查 `metadata.len()`，超过阈值直接拒绝并返回错误提示；读取后检查字符数，超过阈值截断并追加提示
- 优点: 实现简单，两层防护（字节级拒绝 + 字符级截断）
- 缺点: 无
- 工作量估算: S

### 方案 2: 分块读取 + 流式截断（理想架构）

- 思路: 用 `tokio::io::BufReader` 分块读取，达到字符上限后停止，避免大文件完全加载到内存
- 优点: 即使文件恰好在阈值附近也不会多读
- 缺点: 实现复杂度高，需处理 UTF-8 多字节边界；对于阈值内的文件（绝大多数场景）反而更慢
- 工作量估算: M

### 推荐

方案 1。预读 metadata 检查已足够防止 OOM（拒绝 > 512KB 的文件），字符截断处理边界情况。分块读取的复杂度不值得。

## 功能需求列表

### 核心功能

- 在 `ReadFileTool::execute()` 中，`read_to_string` 之前检查 `metadata.len()`，超过 `MAX_CHARS * 4`（512KB）时返回错误，提示文件过大并引导使用 exec tool 的 head/tail/grep
- 读取后检查 `content.len()` 是否超过 `MAX_CHARS`（128,000 字符），超过则截断到 `MAX_CHARS` 并追加截断提示信息
- 截断提示包含：实际文件大小、已读取字符数、引导使用 exec tool 处理大文件

### 扩展功能

- 无

## 非功能需求

- **性能**：对正常大小文件（< 512KB）无额外开销（metadata 已在现有流程中获取）
- **测试要求**：新增单元测试覆盖三种场景：正常文件、超大文件拒绝、截断文件

## 边界与不做事项

- 不修改 WriteFileTool、EditFileTool、ListDirTool
- 不添加可配置的大小限制参数（硬编码常量即可）
- 不实现分块读取

## 假设与约束

- **技术假设**：`MAX_CHARS * 4`（512KB）作为字节阈值足够覆盖 UTF-8 最坏情况（1 char = 4 bytes）
- **常量值**：`MAX_CHARS = 128_000`，与上游保持一致

## 待确认事项

- 无
