# 实施计划

- [x] 1. 定义健康检查服务配置
   - 在 `GatewayConfig` 中添加 `health_check_port: Option<u16>` 字段
   - 配置为可选字段，无默认值（未配置则不启动服务）
   - 添加配置验证（端口必须大于 0）
   - _需求：2.1、2.2_

- [x] 2. 实现极简 HTTP 健康检查服务核心逻辑
   - 创建 `crates/cli/src/commands/gateway/health_check.rs` 模块
   - 使用 tokio 异步 API `tokio::net::TcpListener` 实现 TCP 监听
   - 实现 HTTP 请求行的最小化解析（仅读取请求行，忽略头部和请求体）
   - 构造并返回 HTTP/1.0 200 响应（状态行 + `OK` 响应体）
   - 暴露单一的 `serve(port: u16)` 异步函数
   - _需求：1.3、1.4、4.1、4.2、4.3_

- [x] 3. 添加命令行参数支持
   - 在 gateway 命令的 `GatewayCmd` 结构体中添加 `--health-check-port` 参数（类型为 `Option<u16>`）
   - 实现命令行参数与配置文件的优先级合并逻辑（命令行参数优先）
   - _需求：2.3_

- [x] 4. 集成健康检查服务到 gateway 命令
   - 在 gateway 命令启动时，检查 `health_check_port` 是否配置
   - 若已配置，使用 `tokio::spawn` 在后台启动健康检查服务
   - 若未配置，跳过健康检查服务启动
   - 健康检查服务随 tokio runtime 关闭而关闭
   - _需求：1.1、1.2、3.1、3.2_

- [x] 5. 实现错误处理和日志记录
   - 健康检查服务启动失败时记录错误日志，但不影响主服务运行
   - 端口被占用时记录错误日志并继续运行主服务
   - 添加健康检查服务启动成功的日志
   - _需求：1.5、3.3_

- [x] 6. 更新服务状态显示
   - 在服务启动状态中显示健康检查服务的启用状态
   - 已配置：显示 `✓ 健康检查服务: 已启用 (端口: {port})`
   - 未配置：显示 `✓ 健康检查服务: 未配置`
