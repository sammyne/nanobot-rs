# 需求文档

## 引言

本项目需要设计一条 GitHub Actions 工作流，用于自动化版本标签管理。当开发者提交包含特定格式 `bump v{version}` 的提交消息时，系统应自动解析版本号并创建对应的 Git tag，然后推送到远程仓库。这将简化版本发布流程，减少手动操作，并确保版本标签与代码提交的一致性。

## 需求

### 需求 1：提交消息检测

**用户故事：** 作为一名开发者，我希望系统能够自动识别包含版本发布意图的提交消息，以便无需手动创建 tag。

#### 验收标准

1. WHEN 提交消息包含 `bump v{version}` 模式时 THEN 系统 SHALL 触发自动打 tag 流程
2. IF 提交消息不包含 `bump v{version}` 模式 THEN 系统 SHALL 不执行任何 tag 操作
3. WHEN 检测到多个 `bump v{version}` 模式时 THEN 系统 SHALL 只处理第一个匹配项

### 需求 2：版本号解析与验证

**用户故事：** 作为一名开发者，我希望系统能够正确解析和验证版本号，以便生成的 tag 格式正确且有效。

#### 验收标准

1. WHEN 提交消息包含有效的版本号格式（如 `v1.0.0`、`v2.1.3`）THEN 系统 SHALL 成功提取版本号
2. IF 版本号格式无效（如 `v1.x`、`v123`）THEN 系统 SHALL 记录错误日志并终止流程
3. IF 提取的版本号已存在于仓库的 tag 列表中 THEN 系统 SHALL 记录警告日志并终止流程，避免重复创建

### 需求 3：Tag 创建与推送

**用户故事：** 作为一名开发者，我希望系统能够自动创建 tag 并推送到远程仓库，以便触发后续的发布流程。

#### 验收标准

1. WHEN 版本号验证通过 THEN 系统 SHALL 创建一个轻量级 tag，名称为提取的版本号（如 `v1.0.0`）
2. WHEN tag 创建成功 THEN 系统 SHALL 将该 tag 推送到远程仓库
3. IF tag 推送失败（如网络错误、权限不足）THEN 系统 SHALL 重试最多 3 次，并在失败后发送通知

### 需求 4：工作流触发条件

**用户故事：** 作为一名开发者，我希望工作流仅在特定分支上触发，以便保护主分支的稳定性。

#### 验收标准

1. WHEN 提交推送到 `main` 分支且消息包含 `bump v{version}` THEN 系统 SHALL 触发工作流
2. IF 提交推送到非 `main` 分支 THEN 系统 SHALL 不触发工作流
3. WHEN 配置需要时 THEN 系统 SHALL 支持扩展为其他指定分支（如 `master`、`release/*`）

### 需求 5：错误处理与日志

**用户故事：** 作为一名开发者，我希望工作流在出错时能够提供清晰的日志信息，以便快速定位和解决问题。

#### 验收标准

1. WHEN 工作流执行过程中发生错误 THEN 系统 SHALL 在 GitHub Actions 日志中输出详细的错误信息
2. WHEN 工作流成功完成 THEN 系统 SHALL 输出创建的 tag 名称和对应的 commit SHA
3. IF 权限不足导致 tag 推送失败 THEN 系统 SHALL 提示检查 `GITHUB_TOKEN` 权限配置
