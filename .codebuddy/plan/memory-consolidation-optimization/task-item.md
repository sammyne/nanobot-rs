# 实施计划

- [ ] 1. 重构 process_message 方法结构
   - 在 `process_message` 中添加异步整合任务启动逻辑
   - 在 `build_messages` 调用前检查整合条件并启动异步任务
   - 返回 `JoinHandle` 供后续等待使用
   - _需求：1.1、1.2、1.3_

- [ ] 2. 实现整合条件检查逻辑
   - 检查消息数量是否达到整合阈值
   - 使用 `consolidating` Mutex 检查是否有进行中的整合任务
   - 根据检查结果决定是否启动整合任务
   - _需求：1.2、1.3、4.1_

- [ ] 3. 实现异步整合任务启动
   - 使用 `tokio::spawn` 启动记忆整合任务
   - 在任务中正确传递必要的资源（provider、context 等）
   - 在任务启动前标记会话为正在整合状态
   - _需求：1.2、4.1、4.3_

- [ ] 4. 在 re_act 后添加整合等待逻辑
   - 在 `re_act` 完成后检查 `JoinHandle` 是否存在
   - 使用 `await` 等待整合任务完成
   - 处理整合任务的结果（成功/失败）
   - _需求：2.1、2.2、2.4_

- [ ] 5. 更新会话的 last_consolidated 字段
   - 当整合任务成功完成时，更新 `last_consolidated` 值
   - 确保更新操作在 `save_turn` 之前完成
   - _需求：2.3_

- [ ] 6. 调整 save_turn 的执行时机
   - 将 `save_turn` 调用移到整合等待逻辑之后
   - 确保在整合完成（或跳过整合）后再保存消息
   - _需求：3.1、3.2_

- [ ] 7. 实现整合状态清理机制
   - 在整合任务完成（无论成功或失败）后，从 `consolidating` HashSet 中移除会话标识
   - 确保在异步任务中使用 RAII 模式或 finally 逻辑保证状态清理
   - _需求：4.2、5.5_

- [ ] 8. 添加日志记录
   - 在整合任务启动时记录日志
   - 在等待整合完成时记录日志
   - 在整合成功/失败时记录相应的日志
   - _需求：5.1、5.2、5.3、5.4_

- [ ] 9. 编写单元测试
   - 测试整合任务在阈值达到时正确启动
   - 测试整合任务在阈值未达到时不启动
   - 测试整合任务在有进行中任务时不启动
   - 测试整合等待逻辑正确处理成功和失败情况
   - _需求：1.2、1.3、2.2、2.4_

- [ ] 10. 进行集成测试和性能验证
   - 测试完整的消息处理流程
   - 验证整合与 LLM 调用的并行执行效果
   - 验证数据一致性（last_consolidated 正确更新）
   - _需求：1.1、2.3、3.1_
