# 实施计划：Heartbeat Component

## 概述
本实施计划基于《需求文档：Heartbeat Component》，将12个功能性需求和4个非功能需求转化为可执行的编码任务清单。任务按照依赖关系和实施顺序组织，从基础结构到核心功能，再到集成和测试。

## 任务清单

- [x] 1. 创建 heartbeat crate 基础结构
  - 在 `crates/` 下创建 heartbeat 目录和标准的 Rust crate 结构（src/lib.rs, Cargo.toml）
  - 添加必要的依赖项：tokio, serde, tracing, thiserror, async-trait
  - 在 Cargo workspace 的根 Cargo.toml 中注册新成员
  - 定义基础的模块结构（config, service, error, callback）
  - _需求：非功能需求-可维护性需求（代码风格）_

- [x] 2. 实现核心数据模型和配置结构
  - 创建 `HeartbeatConfig` 结构体，包含 enabled 和 interval_seconds 字段
  - 实现默认配置（enabled: true, interval_seconds: 1800）
  - 添加配置验证逻辑（确保 interval_seconds > 0）
  - 实现 serde 的序列化和反序列化支持
  - 创建 `HeartbeatError` 枚举类型，使用 thiserror 定义错误变体
  - _需求：9.1, 9.2, 9.3, 9.4, 11.1, 11.2, 11.3, 11.4, 11.5, 11.8, 11.9, 非功能需求-可维护性需求（错误处理）_

- [x] 3. 定义回调函数接口
  - 创建 `OnExecuteCallback` 类型定义（async trait 或函数指针）
  - 创建 `OnNotifyCallback` 类型定义
  - 定义回调函数的签名，包括输入参数（任务摘要）和返回值（结果字符串）
  - 实现 Option 类型的回调字段支持
  - _需求：4.1, 4.2, 4.4, 5.1, 5.3, 5.4_

- [x] 4. 实现 HeartbeatService 结构体和生命周期管理
  - 创建 `HeartbeatService` 结构体，包含配置、路径、provider、模型名称、回调等字段
  - 实现构造函数，接受所有依赖注入参数
  - 实现启动方法，处理并发启动保护
  - 实现停止方法，清理异步任务句柄并确保状态一致性
  - 添加运行状态标志，防止重复启动
  - _需求：8.1, 8.2, 8.3, 8.4, 10.1, 10.2, 10.3, 10.4, 10.5_

- [x] 5. 实现周期性心跳检查机制
  - 使用 tokio::time::interval 创建定时器
  - 实现心跳检查循环，在配置间隔时触发检查
  - 实现启动时检查 enabled 配置，禁用时不启动定时器
  - 添加异常处理，确保单个失败不中断循环
  - 实现服务停止时取消定时任务
  - _需求：1.1, 1.2, 1.3, 1.4, 1.5, 非功能需求-可靠性需求（自动恢复）_

- [x] 6. 实现两阶段决策机制 - Phase 1（决策阶段）
  - 创建 `check` 方法执行完整的心跳检查流程
  - 实现读取工作区中 HEARTBEAT.md 文件的逻辑
  - 处理文件不存在或为空的边界情况
  - 构建虚拟 heartbeat 工具定义（action 参数，枚举值 skip/run）
  - 调用 LLM provider 发送工具调用请求
  - 解析 LLM 响应，提取 action 和 tasks 参数
  - 处理 LLM 响应不包含工具调用的情况（视为 skip）
  - _需求：2.1, 2.2, 2.3, 2.4, 2.5, 2.6, 3.1, 3.2, 3.5, 3.6_

- [x] 7. 实现两阶段决策机制 - Phase 2（执行阶段）
  - 实现 `execute_phase` 方法，在 action="run" 时触发
  - 调用配置的 `on_execute` 回调函数
  - 检查回调返回结果，处理空结果情况
  - 如果回调返回非空结果，调用 `on_notify` 回调函数
  - 实现结果日志记录（info 级别记录 skip 和执行情况）
  - _需求：2.7, 4.2, 4.3, 4.4, 5.2, 5.3, 5.4_

- [x] 8. 实现手动触发功能
  - 创建 `trigger` 方法，支持手动触发心跳检查
  - 手动触发时执行完整的 Phase 1 和 Phase 2 流程
  - 返回适当的执行结果（执行结果字符串或 None）
  - 集成异常处理和日志记录
  - _需求：6.1, 6.2, 6.3, 6.4, 7.3, 7.4, 7.5, 7.6_

- [x] 9. 集成网关配置和工作空间模板
  - 在 GatewayConfig 结构体中添加 `heartbeat: HeartbeatConfig` 字段
  - 更新配置加载逻辑，正确解析 gateway.heartbeat 配置节
  - 在 onboard 命令中定义默认 HEARTBEAT.md 文件内容模板
  - 实现写入 HEARTBEAT.md 到新工作空间的逻辑
  - 添加文件存在性检查，避免覆盖现有文件
  - 实现错误处理（返回 OnboardError::FileWrite）
  - _需求：11.6, 11.7, 12.1, 12.2, 12.3, 12.4, 12.5, 12.6, 12.7, 12.8_

- [x] 10. 编写单元测试和集成测试
  - 为配置模型和验证逻辑编写单元测试
  - 为 HeartbeatService 生命周期管理编写测试（启动/停止/并发）
  - 模拟文件系统和 LLM provider，测试两阶段决策逻辑
  - 测试回调函数的调用和参数传递
  - 测试异常场景和边界情况
  - 编写手动触发功能的测试
  - 编写集成测试，验证完整的端到端流程
  - 使用表驱动测试和 Arrange-Act-Assert 模式
  - _需求：非功能需求-可测试性需求（mock 和抽象）, 非功能需求-可维护性需求（测试分离）_

## 依赖关系说明

- 任务 2 必须在任务 1 之后完成
- 任务 3 必须在任务 4 之前完成（回调接口定义）
- 任务 4 是所有后续任务的基础
- 任务 5、6、7 是核心功能，相互独立但都需要任务 4
- 任务 8 依赖任务 6 和 7 的完成
- 任务 9 可以独立进行，但依赖任务 2 的配置结构
- 任务 10 应在所有功能任务完成后进行
